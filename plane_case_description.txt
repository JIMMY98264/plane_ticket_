案例思路：
- 这是一个基于二进制文件的简单机票预订演示程序。通过 `Airline` 类收集用户输入（姓名、出发日期、出发地、目的地、航班号、状态），将对象按二进制块追加写入 `plane.txt`。程序提供四个操作：新建预定、查看所有预定、修改某条预定的状态、退出。

核心要点：
- 二进制序列化：使用 `ofstream::write` / `ifstream::read` 直接按对象内存写入/读取记录，方便追加和按记录大小定位。
- 固定长度字段：类中用固定大小的 `char[]` 存储字符串，便于定长记录的读写和随机访问。
- 搜索与更新：按预定号（`prn`）线性扫描文件，找到匹配记录后用 `seekp(-sizeof(a), ios::cur)` 回退并覆盖实现就地修改。
- 简单交互式菜单：通过 `cin` 获取用户选择并执行对应分支。
- 资源管理：用 `fstream` / `ifstream` / `ofstream` 打开并在操作完成后关闭文件。

常见问题与解答（FAQ）：
Q1：为什么要把二进制对象写入替换为文本？
A1：二进制按内存写入依赖于对象内存布局（填充、编译器、结构对齐），对字符串等非平凡成员尤其危险，会导致溢出与不可移植性。文本序列化更透明、易调试、跨平台且不依赖编译器实现细节。

Q2：改为 `std::string` 就完全安全了吗？
A2：`std::string` 本身更安全（管理内存、可变长），但仍需正确读取（使用 `getline`）和验证输入长度/格式，并处理 I/O 错误。文本序列化后需处理分隔符和转义问题。

Q3：为什么修改时先加载所有记录再重写文件？性能如何？
A3：这种方式避免了二进制就地覆盖的结构依赖和复杂的字节偏移计算，简化实现并降低风险。对于小型数据集性能开销可忽略；若记录很多，可改用索引、内存映射或数据库（SQLite）来提高性能。

Q4：如何保证 `plane.csv` 不被破坏（部分写入）？
A4：可采用写临时文件并在写入成功后重命名原理（原子替换），或者在写入时使用文件锁，并在每条记录前写入校验和/版本号用于恢复。

Q5：如何处理字段包含分隔符 `|` 的情况？
A5：本实现用 `sanitize_field` 将 `|` 替换为 `/`。更严格的方法是使用 CSV（带引号并按 RFC 4180 转义）或 JSON，以支持任意字符并用成熟库处理解析/序列化。

附加练习问题与答案（供面试或学习参考）：
问：如果要扩展为支持按预定号快速查找，应如何修改存储结构？
答：可以在内存或磁盘上维护一个索引（哈希表或 B 树），将 `prn` 映射到记录行号或文件偏移；更新时同步更新索引。对于持久化和并发建议使用 SQLite 或键值数据库（LevelDB等）。

问：为什么在 `setStatue()` 中使用 `char` 而不是 `bool` 或 `enum`？应该怎么改？
答：原始实现使用 `char` 简单记录字符状态；更严谨的做法是定义 `enum class Status { Pending, Confirmed, Cancelled }` 并序列化为稳定的字符串或整数，便于扩展与可读性。

问：如何添加输入校验？
答：对 `prn` 使用正则表达式（例如 `^[A-Z0-9]{4,8}$`），对日期采用 `YYYY-MM-DD` 格式并用日期解析函数验证合法性，对姓名/城市做长度限制并过滤非法字符。

对先前面试题的答案（逐题）：
1) 为什么选择二进制写对象 vs 文本（CSV/JSON）？
	- 答：二进制写对象优点是简单、写入/读取速度快（无需格式化），缺点是不可移植、受编译器/内存布局影响、难以查看和调试、易受缓冲区溢出影响。文本格式（CSV/JSON）优点是可读、可移植、易于与其他工具交互、易于逐字段验证；缺点是解析/序列化成本更高，文件会更大。

2) 固定大小 `char` 数组的问题及改进方法？
	- 答：固定大小 `char[]` 容易出现缓冲区溢出、需手动管理终止符、难以处理超过长度的输入，且降低可维护性。改进：使用 `std::string`、使用 `strncpy` 并强制终止符、使用安全输入函数、或采用文本序列化并检查字段长度。

3) `prn` 唯一性与索引设计？
	- 答：最简单是添加在写入前扫描文件查重（O(n)），更好的做法是维护一个索引（内存哈希或磁盘索引），或者使用数据库约束（SQLite 的 UNIQUE 索引）。索引可以基于 `prn` 映射到行号或文件偏移量以实现 O(1) 或 O(log n) 查找。

4) `seekp(-sizeof(a), ios::cur)` 的原理和风险？
	- 答：该操作基于固定大小记录的假设，回退写入前一条记录进行覆盖。风险包括：对象大小/布局随编译器或代码改变而变化；字符串或指针成员的内存不能直接二进制覆盖；如果写失败或中断可能导致文件损坏。文本格式或数据库更新更安全。

5) 并发读写会带来的问题与改造方案？
	- 答：并发会引发竞态、部分写入、丢失更新和读到半写数据等问题。改造方案：使用进程/线程级文件锁（fcntl、LockFileEx 等）、采用写临时文件再原子替换、或迁移到支持并发和事务的数据库（SQLite 在开启 WAL 模式下也支持多读单写）。
